ДОКЛАД К СЛАЙДУ 2: Архитектура хранения и базовые требования (Задание 0)
--------------------------------------------------------------------------------
"Перейдем к описанию физического уровня хранения данных и базовых требований СУБД.
Хранилище спроектировано как строго типизированное и имеет иерархическую структуру. 
На диске база данных представлена директорией, внутри которой для каждой таблицы 
создается два типа файлов: файл данных и файл индексов.
Мы поддержали модификаторы NOT_NULL и INDEXED. Модификатор INDEXED накладывает 
условие уникальности поля и запрещает хранение NULL-значений. Для таких полей в 
СУБД автоматически строится индекс на основе структуры B+ Дерева (B-Plus Tree). 
Важная деталь реализации: дублирование данных записей в индексе полностью исключено. 
Узлы B+ дерева хранят упорядоченные ключи и числовые смещения (offset) соответствующих 
записей в основном файле таблицы на диске.

Парсинг SQL-запросов реализован на базе генератора лексических анализаторов Flex 
и генератора LALR-парсеров Bison. На этапе разбора строится Query Plan — дерево 
выполнения запроса. Если в запросе SELECT присутствует условие WHERE по индексированной 
колонке, оптимизатор запросов автоматически исключает полный скан таблицы (Full Table Scan) 
и перенаправляет поиск через B+ дерево, выполняя операцию поиска за логарифмическое 
время O(log N).
СУБД работает в двух режимах: интерактивный консольный терминал REPL и пакетный режим, 
в котором последовательно выполняются SQL-команды из переданного `.sql` файла."

ДОКЛАД К СЛАЙДУ 3: Сетевая архитектура "Клиент-Сервер" (Задание 3)
--------------------------------------------------------------------------------
"Для обеспечения возможности сетевой работы и масштабирования СУБД была разделена 
на два изолированных компонента: клиентский терминал и многопоточный сервер.

Основной функционал СУБД вынесен в серверное приложение. Сетевой слой реализован 
на базе многопоточного C++ HTTP-фреймворка Crow. Сервер слушает входящие TCP-соединения 
и обрабатывает запросы пулом потоков. Клиентское приложение представляет собой легкий 
терминал `dbms_client.exe`, написанный с использованием библиотеки `httplib`. 
Клиент отвечает только за чтение ввода пользователя, формирование JSON-тела запроса, 
отправку его по сетевому сокету на сервер (метод HTTP POST на эндпоинт `/query`) 
и последующий красивый вывод полученного структурированного ответа. 
Обмен данными стандартизован в формате JSON, что позволяет при необходимости подключить 
к нашему серверу любой сторонний клиент или веб-интерфейс."

ДОКЛАД К СЛАЙДУ 4: Кластеризация и Шардирование данных (Задание 4)
--------------------------------------------------------------------------------
"Для решения проблемы горизонтального масштабирования была реализована кластерная 
архитектура. Она состоит из двух ключевых компонентов:
1. Балансировщик-координатор (Entrypoint) — единая точка входа для клиентов.
2. Узлы хранения (Storage Nodes) — изолированные процессы баз данных, непосредственно 
   работающие с диском.

При отправке клиентом SQL-запроса, координатор парсит тело запроса и извлекает имя 
таблицы, к которой происходит обращение. Далее от имени таблицы вычисляется хэш-значение 
по быстрому некриптографическому алгоритму FNV-1a. По значению хэша координатор 
определяет целевой узел (Storage Node) по формуле `index = hash % nodes_count` и 
проксирует запрос на соответствующий физический сервер.
Реализовано динамическое управление топологией кластера: через специальные 
административные HTTP-эндпоинты `/cluster/add_node` и `/cluster/remove_node` можно 
добавлять новые серверы хранения или выводить их из работы прямо во время выполнения 
запросов пользователями, без остановки и перезапуска всей СУБД. Данные при этом 
автоматически перераспределяются."

ДОКЛАД К СЛАЙДУ 5: Отказоустойчивость: Heartbeat Monitor (Задание 5)
--------------------------------------------------------------------------------
"Важнейшим аспектом распределенных систем является отказоустойчивость. Для её обеспечения 
в координатор интегрирован модуль Heartbeat-мониторинга.

В фоновом потоке координатора с настраиваемым интервалом (по умолчанию 5 секунд) 
запущен бесконечный цикл опроса состояния всех зарегистрированных узлов кластера. 
Опрос происходит через легковесные системные запросы PING-PONG. Если один из серверов 
хранения не отвечает в течение заданного таймаута (2 секунды), координатор переводит 
статус узла в RESTARTING и посылает операционной системе команду принудительного запуска 
процесса `storage_server.exe` с параметрами этого узла (порт, хост и рабочая папка). 
После успешного старта процесса координатор восстанавливает TCP-соединение и возвращает 
узел в статус ONLINE. Это предотвращает потерю доступности данных при сбоях 
отдельных серверов."

ДОКЛАД К СЛАЙДУ 6: Асинхронное выполнение запросов (Задание 6)
--------------------------------------------------------------------------------
"Для обработки длительных и ресурсоемких аналитических операций, таких как агрегация 
больших объемов данных или построение сложных выборок, была спроектирована асинхронная 
модель обработки на базе потокобезопасной очереди задач (ThreadPool).

При отправке SQL-запроса на асинхронный эндпоинт `/query_async`, координатор не ждет 
завершения выполнения запроса. Он мгновенно регистрирует задачу в очереди, генерирует 
уникальный идентификатор запроса по стандарту GUID версии 4 и возвращает его клиенту 
вместе со статусом `queued` (в очереди) и HTTP-кодом `202 Accepted`.
В фоне пул рабочих потоков (Async Workers) выбирает задачи из очереди, выполняет их 
на целевых шардах и сохраняет результат в потокобезопасном хранилище результатов. 
Клиент может периодически опрашивать сервер методом GET по адресу `/async/<guid>` для 
получения текущего статуса задачи (`queued` -> `running` -> `completed` / `failed`) и 
забора готового набора данных."

ДОКЛАД К СЛАЙДУ 7: Безопасность: Аутентификация и RBAC (Задание 9)
--------------------------------------------------------------------------------
"Безопасность распределенной СУБД обеспечивается подсистемой разграничения 
доступа на основе ролей (RBAC) и JWT-аутентификацией.

Все пароли пользователей в файле `accounts.json` защищены: они хэшируются с солью 
с использованием многораундового хэш-алгоритма, что предотвращает атаки по словарям 
и «радужным таблицам». При входе на эндпоинт `/login` сервер проверяет учетные данные 
и возвращает сгенерированный JWT-токен, подписанный секретным ключом сервера. В токене 
закодированы имя пользователя и время жизни токена (exp).
Каждый последующий запрос к эндпоинтам `/query` или `/query_async` требует наличия 
токена в заголовке `Authorization: Bearer <token>`.
При разборе запроса сервер извлекает роль и имя пользователя и сопоставляет их с 
матрицей прав доступа конкретной БД. Разграничиваются следующие права: чтение (SELECT), 
запись (INSERT, UPDATE, DELETE), создание таблиц (CREATE TABLE), удаление таблиц 
(DROP TABLE) и удаление базы данных (DROP DATABASE). Если операция запрещена, сервер 
возвращает HTTP-код `403 Forbidden`."

ДОКЛАД К СЛАЙДУ 8: Логирование активности и Телеметрия (Задания 7 и 8)
--------------------------------------------------------------------------------
"Для непрерывного мониторинга запущенной СУБД и расследования инцидентов безопасности 
были реализованы модули Access Logs и Сбора Телеметрии.

Подсистема Access Logs работает асинхронно через отдельный фоновый поток-писатель, 
чтобы дисковые операции записи логов не замедляли обработку SQL-запросов. В лог-файл 
`server_access.log` записывается каждая операция: временная метка начала запроса, 
ID клиента (имя пользователя), уникальный ID обработчика (метка `[sync]` или сгенерированный 
GUID для асинхронных задач), статус выполнения (OK с кодом `200`/`202` или ERR с кодом 
ошибки, например `403`/`400`), время обработки в миллисекундах, экранированное тело 
запроса и детальное описание ошибки, если она возникла.
Параллельно модуль телеметрии собирает метрики производительности в скользящем временном 
окне. Через API `/cluster/stats` в реальном времени можно получить: текущий RPS, 
средний и максимальный RPS за последние 10 минут, среднее время обработки запросов 
за последние 10 секунд и коэффициент ошибок (Error Rate) за минуту."

ДОКЛАД К СЛАЙДУ 9: Дополнительные оптимизации и фичи (Задания 1, 2, 10, 11, 12)
--------------------------------------------------------------------------------
"Помимо распределенной архитектуры, в ядро СУБД был внесен ряд важных математических 
и алгоритмических улучшений.

Во-первых, реализован механизм темпоральной персистентности (команда `REVERT`). 
СУБД ведет журнал транзакций (Write-Ahead Log / WAL). При вызове команды REVERT 
система анализирует временные метки операций в журнале и последовательно откатывает 
или накатывает изменения записей до указанного пользователем момента времени с точностью 
до миллисекунд. Полное копирование файлов (снапшоты) не производится, что экономит память.

Во-вторых, внедрена дедупликация строк (String Interning/Pool) в оперативной памяти. 
Все текстовые литералы хранятся в единственном экземпляре в глобальном пуле строк, 
а в таблицах используются лишь легковесные указатели-индексы на них, что радикально 
снижает потребление ОЗУ при обработке больших объемов текстовых данных.

В-третьих, реализована поддержка значений по умолчанию (Задание 10). Синтаксис `CREATE TABLE` 
был расширен модификатором `DEFAULT [value]`. При добавлении записей через команду `INSERT`, 
если для какого-то столбца значение не передано явно, ядро СУБД автоматически подставляет 
настроенное значение по умолчанию из метаданных схемы таблицы (для типов `INT` и `STRING`).

Также мы расширили возможности языка запросов SQL: парсер поддерживает приоритеты 
операций AND и OR и круглые скобки в WHERE, а в выборках SELECT поддерживаются агрегаты 
SUM, COUNT и AVG."
