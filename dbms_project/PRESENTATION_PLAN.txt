СЛАЙД 2: Архитектура хранения и базовые требования (Задание 0)
--------------------------------------------------------------------------------
* Заголовок: Архитектура СУБД и Язык Запросов
* Содержимое слайда:
  - Двухуровневая структура: Уровень Систем (БД) -> Уровень БД (Таблицы) -> Уровень Таблиц (Записи).
  - Поддержка модификаторов колонок: NOT_NULL и INDEXED.
  - Индексация: B+ Деревья для индексированных полей (ускоряет WHERE поиск).
  - Вывод результатов: Форматированный массив JSON.
  - Режимы запуска: Интерактивный терминал и пакетный скриптовый режим.
* Тезис для доклада: "СУБД хранит все данные в файловой системе. Нами реализована
  синтаксическая валидация SQL через Flex/Bison, поддержка B+ деревьев для индексов,
  интерактивный терминал и пакетная обработка скриптов."

--------------------------------------------------------------------------------
СЛАЙД 3: Сетевая архитектура "Клиент-Сервер" (Задание 3)
--------------------------------------------------------------------------------
* Заголовок: Клиент-Серверное Взаимодействие
* Содержимое слайда:
  - Вынесение ядра СУБД в многопоточный HTTP-сервер на базе Crow.
  - Легковесный клиент-терминал (`dbms_client.exe`), общающийся по сокетам.
  - Схема взаимодействия: Клиент (SQL) -> HTTP POST -> Сервер (Engine) -> JSON-ответ -> Клиент.
* Тезис для доклада: "Для масштабирования СУБД была вынесена в сетевой сервис.
  Клиент отправляет SQL-запросы в теле HTTP POST-запросов, а ядро выполняет парсинг,
  семантическую валидацию и возвращает структурированный JSON."

СЛАЙД 4: Кластеризация и Шардирование данных (Задание 4)
--------------------------------------------------------------------------------
* Заголовок: Распределенная архитектура и Шардирование
* Содержимое слайда:
  - Архитектура: Узел-координатор (Entrypoint) и несколько серверов хранения (Storage Nodes).
  - Шардирование: Равномерное распределение таблиц по узлам через хэш-функцию FNV-1a.
  - Динамическое масштабирование: Подключение и отключение узлов без остановки (API: /cluster/add_node).
* Тезис для доклада: "В расширенном режиме СУБД работает как кластер. Координатор
  принимает запросы, вычисляет хэш от имени таблицы и перенаправляет запросы на
  нужный узел хранения, распределяя нагрузку."

--------------------------------------------------------------------------------
СЛАЙД 5: Отказоустойчивость: Heartbeat Monitor (Задание 5)
--------------------------------------------------------------------------------
* Заголовок: Мониторинг доступности узлов (Heartbeat)
* Содержимое слайда:
  - Периодический опрос узлов координатором (раз в 5 секунд).
  - Автоматическое обнаружение сбоев (Node Offline).
  - Механизм восстановления: Принудительный автоматический перезапуск процесса
    Storage-сервера координатором при потере связи.
* Тезис для доклада: "Чтобы кластер был отказоустойчивым, реализована фоновая
  проверка доступности (Heartbeat). При падении узла координатор самостоятельно
  перезапускает его процесс и переподключается к нему."

СЛАЙД 6: Асинхронное выполнение запросов (Задание 6)
--------------------------------------------------------------------------------
* Заголовок: Асинхронная очередь запросов
* Содержимое слайда:
  - Очередь запросов для долгих операций (не блокирует клиента).
  - Мгновенный ответ с выдачей уникального идентификатора GUID v4.
  - API для проверки статуса выполнения (`queued`, `running`, `completed`, `failed`)
    и последующего получения результата.
* Тезис для доклада: "Для предотвращения зависания клиентов при тяжелых запросах
  реализована асинхронная очередь. Сервер мгновенно выдает GUID v4, а клиент
  в фоновом режиме может отслеживать прогресс выполнения задачи."

СЛАЙД 7: Безопасность: Аутентификация и RBAC (Задание 9)
--------------------------------------------------------------------------------
* Заголовок: Разграничение доступа (RBAC) и JWT
* Содержимое слайда:
  - Аутентификация: Выдача JWT-токенов пользователям при входе (/login).
  - Безопасное хранение аккаунтов: Пароли хранятся в виде криптографических хэшей с солью.
  - Ролевой доступ (RBAC): Настройка прав (Read, Write, Create/Drop Table, Drop DB)
    для пользователей и групп на уровне конкретной базы данных.
* Тезис для доклада: "Безопасность реализована через аутентификацию по JWT-токенам.
  Пароли хэшируются с солью. Права доступа настраиваются индивидуально или для групп
  на уровне баз данных, проверяясь при каждом SQL-запросе."

  доступная через REST API в реальном времени."

СЛАЙД 9: Дополнительные оптимизации и фичи (Задания 1, 2, 10, 11, 12)
--------------------------------------------------------------------------------
* Заголовок: Дополнительные возможности ядра
* Содержимое слайда:
  - Темпоральная персистентность (REVERT): Откат состояния таблиц к указанной дате без создания снапшотов.
  - Дедупликация строк (String Interning): Оптимизация ОЗУ (уникальные строки хранятся в пуле).
  - Значения по умолчанию (DEFAULT): Поддержка модификатора `DEFAULT` в `CREATE TABLE` и автоматическая вставка при `INSERT`.
  - Сложная алгебра логики: Поддержка AND, OR и группирующих скобок `( ... )` в условиях WHERE.
  - Агрегаты в SELECT: SUM, COUNT, AVG.
* Тезис для доклада: "В ядре СУБД оптимизировано хранение строк через пул дубликатов,
  реализован откат таблиц по времени (REVERT) без перегрузки диска, добавлена поддержка
  значений по умолчанию (DEFAULT) при вставках, а также поддерживаются сложные логические
  выражения WHERE и агрегаты (SUM, COUNT, AVG)."
